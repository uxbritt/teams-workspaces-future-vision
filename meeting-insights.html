  • Define "North Star" first

  • "Switch Teams" is heavyweight

  • Dual-action rows are confusing

    "Requiring users to pick between two actions on one row is complicated if they don't understand the terms."

  • ROI analysis required

    "The goal is to make the experience feel like simple navigation rather than logging in or out."

Key Findings from Meetings

Pain Points Identified

  • Inconsistent terminology ‚Äî Multiple terms (multi-account, parent, partner, child, sub-account) create confusion
  • Limited visibility ‚Äî Customers with 100+ sub-accounts can't see full org structure and get lost
  • 75% use impersonation ‚Äî Required to perform basic tasks, creating friction
  • "Switch Teams" is heavyweight ‚Äî Feels like logging in/out rather than navigating
  • Hub/Workspace dual actions confusing ‚Äî Users don't understand picking between two actions on one row

User Testing Results

  • "Teams tier" understood ‚Äî Users grasped the concept when explained
  • "Team" term mixed ‚Äî 2/4 users thought it meant team members, not accounts
  • Discovery issues ‚Äî Users had trouble finding structure initially, but found it convenient once discovered
  • Only 4 users tested ‚Äî More validation needed before major decisions

A Evolutionary Approach

Improve current structure within existing architecture

Core Concept

Keep impersonation mechanism but make it feel like navigation. Improve visibility and simplify terminology while working within current technical constraints.

Key Features

  • Simple "Up/Back" navigation instead of "Switch Teams"
  • Always-visible organization tree structure
  • Clear context indicator showing current view (Hub vs Workspace)
  • Different nav sections for Hub actions vs Workspace actions
  • Impersonation hidden behind seamless navigation

Pros

  • Can be implemented faster with existing architecture
  • Lower technical risk and development cost
  • Addresses navigation and visibility pain points

Cons

  • Doesn't solve underlying roles/permissions architecture
  • May delay necessary fix to complex workflow (per meeting concern)
  • Technical debt accumulates

B North Star Approach

True multi-account architecture without impersonation

Core Concept

Solve the roles/permissions issue architecturally. One user can access multiple accounts natively. Hub is purely for cross-account functions. Corporate sending moves to child workspace.

Key Features

  • Split navigation: Top bar for Hub, Side bar for Workspace
  • Direct workspace access‚Äîno impersonation needed
  • Corporate office as child workspace (Lee Ott's suggestion)
  • File-system style workspace navigation
  • Hub actions always accessible in top bar

Pros

  • Solves root cause of friction (roles/permissions)
  • Eliminates 75% impersonation need
  • Scales better for customers with 100+ accounts
  • Clear separation of Hub vs Workspace experiences

Cons

  • Significant architectural investment required
  • Longer development timeline
  • Requires ROI analysis (per Lee Ott request)
